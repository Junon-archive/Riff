# 08 · 산문 하이라이트 + 커리큘럼 볼드 (레시피 동결)

## 목적 (왜)

레슨 산문이 "글의 벽"이라 핵심이 눈에 안 들어온다. **중요한 부분에 연한 하늘색 하이라이트**를 얹어 훑기(scan) 가능하게 한다.

- **각 day 설명 페이지(레슨 산문 ①이론 ②시각 ③연습 ④팁):** 하이라이트만.
- **커리큘럼 설명 페이지(intro 소개 문단):** 하이라이트 **+ 볼드** 둘 다.

**⚠️ 왜 "레시피 동결"인가:** 대상이 **day 116개 × 3언어 = 348파일**(+커리큘럼 intro 3×3언어)로 방대하다. "매 페이지에서 무엇을 강조할지 그때그때 판단"하면 **뒤로 갈수록 기준이 흔들려**(판단 피로) 앞뒤 페이지 강조 밀도·품질이 들쭉날쭉해진다. 그래서 **판단(레시피 수립)과 실행(적용)을 분리**한다 — 작은 파일럿에서 강조 기준을 **명문 규칙으로 동결**한 뒤, 나머지는 그 규칙만 기계적으로 적용한다. 05_update_backlog 공통 정신("나중에 문서만 보고 이어서")과 동일 — 규칙이 문서에 박제돼 있으면 세션이 끊겨도 동일 품질로 재개된다.

**요약 — 두 표면의 준비도가 다르다:**
- **day 페이지:** 렌더 준비 완료. 색만 하늘색으로 바꾸면 **콘텐츠(`<mark>`) 추가만으로** 동작.
- **커리큘럼 페이지:** intro가 이스케이프·미변환·미스타일 → **코드 3곳 선행 필요**(set:html + 인라인 포매터/HTML 저장 + `.curr-intro` mark·b 스타일).

## 수정 상세 (실행 지침)

### Phase 0 — 표기 인프라 (색·렌더 준비, 선행 필수)

색과 렌더를 먼저 갖춰야 이후 콘텐츠에 넣는 `<mark>`가 하늘색으로 보인다.

1. **하늘색 하이라이트 토큰 신설(권장):** `tokens.css`에 `--prose-hl`(라이트/다크) 추가 — 라이트 `#E8F3FF`(=blue-50) 계열, 다크는 저휘도 반투명 파랑. **스코어용 `--highlight`(초록)는 건드리지 않는다**(color_legend SSOT 유지). LessonView `.prose mark` 배경을 `--highlight`→`--prose-hl`로 교체.
   - (대안) `--highlight` 자체를 하늘색으로 바꾸는 안은 color_legend와 의미 충돌 → 미해결 질문 참조.
2. **커리큘럼 intro 렌더 활성화(3곳):**
   - `CurriculumView.astro:52` `<p>{p}</p>` → `<p set:html={…}>`로 전환.
   - intro 문자열을 인라인 포매터로 통과(예: `build-content.mjs`의 `inline()` 재사용해 `**…**`·`<mark>` 처리) **또는** intro에 HTML을 직접 저장(신뢰 소스).
   - `app.css`의 `.curr-intro`에 `mark`(=`--prose-hl`)·`b/strong` 스타일 추가.
3. **검증:** 파일럿 한 구절에 `<mark>`·`**볼드**`를 넣어 day·커리큘럼 양쪽에서 하늘색·굵게가 실제로 렌더되는지 확인(build + 육안).

### Phase 1 — 레시피 수립 (판단, 1회·파일럿에서만)

**파일럿 선정:** 커리큘럼 1개의 intro + 그 커리큘럼의 대표 1주(4 day × 3언어). 여기서만 "무엇을 강조할지"를 사람이 판단하고, 결과를 **아래 레시피 규칙으로 동결**한다.

**동결 레시피(파일럿에서 확정해 이 문서에 박제 — 초안 골격):**
- **하이라이트 대상:** 각 문단/섹션의 **핵심 한 방** — 그 문단이 말하는 결론·행동지시·핵심 개념어. "읽는 사람이 이 문단에서 딱 하나 가져가야 할 것".
- **밀도 상한:** 문단당 **1개(최대 2)**, 섹션당 상한 N개(파일럿에서 확정). **문장·문단 통째 하이라이트 금지**(강조가 강조를 잃음).
- **제외:** 헤딩(이미 강조됨), 이미 `<b>`인 어구, 악보 캡션·링크, 필러 문장.
- **볼드(커리큘럼 intro 전용):** 볼드 = 코스를 정의하는 **명사·용어·기간**(예: "3도 착지", "16비트 모터"). 하이라이트 = 그 문단의 **약속 한 줄**. 볼드와 하이라이트를 **같은 어구에 겹쳐 쓰지 않는다**(택1, 파일럿에서 관계 확정).
- **언어 일관성:** ko/en/ja에서 **의미상 등가 구절**에 동일하게 배치(산문은 언어별 번역이라 바이트 동일 아님 — 등가 매핑). 같은 용어 반복 시 규칙(첫 등장만 등) 확정.
- **Before/After 예시:** 파일럿에서 문단별 적용 예를 만들어 이 문서에 남긴다(집행자가 흉내낼 표준).

### Phase 2 — 실행 (기계적 배치, 판단 재개 금지)

동결된 레시피만 보고 배치 단위로 적용한다. **강조 기준을 다시 판단하지 않는다.**

- **배치 단위:** 커리큘럼별 **주(week) 단위**(4 day × 3언어 = 12파일) 권장 — 프로젝트의 주/월 배치 관례와 정합. 더 촘촘히 원하면 day(3파일) 단위.
- **각 배치 절차:** 레시피대로 스캔 → `<mark>`(day) / `<mark>`+`**볼드**`(커리큘럼) 적용 → V1 build → 다음 배치. 3언어 등가 배치 확인.
- **진행 추적:** 아래 체크리스트를 커리큘럼·주 단위로 채운다.

## 기존 사이트/배포 불변 보장

- **콘텐츠 가법:** `<mark>` 삽입은 산문 HTML만 바꾼다. **악보 JSON 블록 불변** → `check-invariants.mjs`(스코어 블록 지문) 회귀 0.
- **day 렌더 무변경:** mark 스타일이 이미 존재 → Phase 0의 색 토큰 교체 외 렌더 코드 불변. mark 없는 기존 문단은 그대로.
- **커리큘럼 렌더:** intro가 없는 커리큘럼(향후)·intro에 강조 없는 경우 모두 기존과 동일하게 렌더(가법 분기).
- **Zero-Cost·i18n·토스톤·계층 구조** 불변([D]군).

## 검증 방법

- **V1** `cd web_app && npm run build` exit 0.
- **V2** 배치 편집 시 ko/en/ja 산문 강조가 등가 구절에 정합(개수·위치 대응). 악보 JSON 블록 3언어 바이트 동일 유지.
- **V3** `node web_app/scripts/check-invariants.mjs` 회귀 0(산문 강조는 스코어 지문에 무관).
- **육안:** day·커리큘럼 페이지에서 하이라이트=하늘색, 커리큘럼 볼드=굵게, 라이트/다크 모두 가독. 밀도 상한 준수(문단당 1~2).

## 주의 / 정지조건

- **색 충돌:** 스코어 `--highlight`(초록, color_legend SSOT)와 산문 하이라이트를 **분리**해야 한다. `--highlight`를 통째로 하늘색으로 바꾸면 스코어 role "highlight"까지 하늘색이 되어 SSOT가 깨진다 → 별도 토큰 필수.
- **커리큘럼 intro는 코드 선행:** set:html 전환 없이 콘텐츠에 `<mark>`/`**`만 넣으면 **글자로 노출**된다. Phase 0을 반드시 먼저.
- **set:html 안전성:** intro는 신뢰 소스(우리 저작)라 XSS 우려는 낮으나, 외부 입력을 intro에 넣지 않는다는 전제 유지.
- **밀도 폭주 금지:** 강조가 많아지면 강조가 아니다. 문단당 상한을 레시피에서 못 박고 집행자가 넘기지 않게.
- **레시피 미동결 상태로 Phase 2 착수 금지:** 판단을 분리한 이유가 무의미해진다. 파일럿·규칙 확정 전엔 실행 배치를 시작하지 않는다.
- **3언어 드리프트:** 한 언어만 강조하고 나머지를 빠뜨리면 정합 깨짐 — 배치는 항상 3언어 묶음으로.

## 미해결 질문 (사람 결정)

- **하늘색 정확한 값** — `#E8F3FF`(blue-50) 그대로 vs 더 연하게/진하게. 다크모드 대응 값.
- **토큰 전략** — 신규 `--prose-hl`(권장, 스코어 초록과 분리) vs `--highlight` 재정의(color_legend까지 갱신해야 함).
- **커리큘럼 intro 저장 형태** — intro에 HTML 직접 저장 vs `inline()` 포매터로 `**…**`·`<mark>` 변환(저작 편의 ↑).
- **하이라이트 밀도 상한** — 문단당 1개 고정 vs 최대 2개 허용. 섹션당 상한.
- **볼드/하이라이트 관계** — 겹침 금지(택1) vs 허용. 커리큘럼에서 볼드=용어/하이라이트=약속 문장 구분 확정.
- **파일럿 범위** — 어느 커리큘럼·어느 주를 대표로 할지.
- **배치 단위** — 주(4day×3언어) vs day(3언어).
